# Koreos v0.2.0 — 6 plateformes, Pong cross-platform, Demo en direct

*30 mai 2026*

Koreos v0.2.0 est disponible sur Maven Central. En cinq sprints, la bibliothèque passe de 3 à **6 plateformes** (macOS, iOS, Android, Windows, Linux, Web) et livre une démo technique complète : **Pong jouable partout avec le même code `commonMain`**. Les démos sont accessibles en direct dans le navigateur.

---

## Ce que v0.2.0 apporte

- **3 nouveaux backends** : Win32 (Windows), X11 + Wayland (Linux), JS + Wasm (Web)
- **9 modules publiés Maven Central** : `koreos-core`, `koreos-appkit`, `koreos-uikit`, `koreos-android`, `koreos-win32`, `koreos-x11`, `koreos-wayland`, `koreos-js`, `koreos-wasm` + façade `koreos`
- **Pong cross-platform** : logique `PongGame.kt` unique tournant sur les 6 plateformes via wgpu4k
- API publique rétro-compatible v0.1.x — seulement des ajouts de variants `RawWindowHandle`

---

## Sprint 3 — Backend Win32 (Windows)

Le backend Windows repose sur **Panama FFM (JDK 25)** via kextract, sans JNI ni JNA. La boucle de messages Win32 supporte les trois modes de `ControlFlow` :

- **Poll** → `PeekMessageW` (non-bloquant, idéal pour le rendu continu)
- **Wait** → `GetMessageW` (bloquant jusqu'au prochain message système)
- **WaitUntil** → `MsgWaitForMultipleObjectsEx` avec timeout calculé

Le `WndProc` est alloué via `Arena.ofShared` — durée de vie égale au processus, jamais libéré, ce qui évite tout use-after-free lorsque Windows rappelle la procédure après la fin du stack Kotlin. La gestion DPI utilise **PerMonitorV2** : `SetProcessDpiAwarenessContext` au démarrage, puis `GetDpiForWindow` à chaque `WM_DPICHANGED` pour recalculer le `scaleFactor` et dispatcher `ScaleFactorChanged`.

## Sprint 2 — Backend Web (JS + Wasm)

La boucle d'événements Web est pilotée par **`requestAnimationFrame`** (RAF) : chaque frame calcule le delta depuis la frame précédente et le passe à `ApplicationHandler.onRedrawRequested`. L'entrée souris et tactile est unifiée dans `DomEventMapper` qui convertit les `PointerEvent` du DOM en `WindowEvent` Koreos — un seul code path pour souris, stylet et touch, sur desktop comme mobile.

Deux modules distincts coexistent : `koreos-js` (Kotlin/JS IR, cible `jsBrowser`) et `koreos-wasm` (Kotlin/Wasm, cible `wasmJsBrowser`), avec une couche commune `koreos-web-common` (lifecycle `pagehide`/`pageshow`, canvas resize via `ResizeObserver`). La façade `koreos` expose les `actual` correspondants sans changement d'API.

## Sprint 4 — Backend Linux (X11 + Wayland)

Le backend Linux détecte automatiquement le compositor actif au démarrage : tentative de connexion Wayland (`wl_display_connect`), fallback X11 si `WAYLAND_DISPLAY` est absent ou la connexion échoue. La variable `KOREOS_LINUX_BACKEND` permet un override explicite.

Les deux backends suivent le **pattern `tryCreate` lazy** : les bindings FFM Xlib et libwayland-client sont chargés dans un `try/catch Throwable` — si la bibliothèque système est absente (ex. Wayland-only sans `libX11.so`), le backend échoue proprement sans crash de la JVM. La boucle Wayland combine `poll` sur le file descriptor `wl_display_get_fd` avec un `eventfd` de réveil (timeout 0 ms en mode Poll, bloquant en mode Wait), ce qui évite le busy-loop tout en restant réactif.

## Sprint 5 — Pong cross-platform

`PongGame.kt` vit entièrement en `commonMain` : pas une ligne de code plateforme-spécifique dans la logique de jeu. Le `GameState` gère balle, raquettes et score en coordonnées normalisées (0.0–1.0). `PongRenderer` utilise wgpu4k pour rendre 5 quads 2D (raquette joueur, raquette IA, balle, deux chiffres de score via bitmap font hardcodée). L'IA (`PongAi`) suit la balle avec un **coefficient de lag** configurable — suffisamment rapide pour être défiée, assez lente pour être battue.

L'entrée est unifiée : flèches clavier sur desktop, zone tactile droite de l'écran sur mobile et web. `requestRedraw` est appelé à chaque `aboutToWait`, ciblant 60 fps sur toutes les plateformes.

---

## Points techniques notables

- **Mode Poll cross-backend** : `PeekMessageW` (Win32) / `XPending` (X11) / `poll fd=0ms` (Wayland) / delta RAF (Web) — même sémantique, implémentations natives
- **Lazy Linux FFM** : `try/catch Throwable` sur le chargement des symboles — aucun crash sur un système Wayland-only sans `libX11`
- **`Arena.ofShared` WndProc** : durée de vie processus, jamais fermé — le callback Win32 reste valide pendant toute l'exécution
